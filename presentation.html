The problem

Small orgs still run on a paper board —
and everyone who didn't walk past it misses out.

Information is trapped

Kitas, clubs, churches pin notices to a cork board. No feed, no calendar, no reminder. Miss the board → miss the closure, the trip, the lice notice.

Digitizing means changing the process

The usual answer is "adopt an app/portal" — a second system one overworked admin must maintain. They won't. So they stay on paper.

The data is sensitive

Notices name children, parents, phone numbers. Pushing photos of a Kita board into a generic cloud tool is a privacy problem these orgs are right to fear.

The orgs least able to adopt software are the ones whose members most need the information to be reachable.

02 · The problemAushang
The one idea

The LLM advises. PII is masked first. A human decides.

Two non-negotiable principles, enforced by architecture — not policy:

1 · Privacy by construction

PII is detected and masked locally, on our own worker, before any external AI call. Raw photos never leave our infra. The LLM only ever sees redacted text — and the provider is swappable (Claude, or EU-hosted Mistral for strict residency).

2 · Advises, doesn't decide

The LLM suggests a content type + dates. Output is schema-validated; an admin confirms before publish. Routing reads only the confirmed value, never the model's.

This is the trust story — and what keeps it out of the EU AI Act's High-risk tier.

How it works

Photograph → OCR → redact → suggest → confirm → publish

📷 Photograph board→ OCR (Tesseract)→ 🔒 Local redact (Presidio)→ LLM suggests (Claude / EU-Mistral)
Schema-validate→ 👤 Admin confirms type→ publish_post (route)→ Feed · Calendar · ICS · Digest

The only text that crosses to the AI is locally-redacted. The only path to a member's screen is an admin pressing publish. Both are enforced, not promised.

04 · The pipelineAushang
Deliverable 2 · No-code POC

Four importable n8n workflows — the pipeline, in no-code

1 · Redaction boundary

Raw text → mask PII → fail-closed assert → only redacted text crosses.

2 · Extract + validate

Claude suggests on redacted text → schema-validate → manual path if invalid.

3 · Confirm + route

Admin corrects the type & publishes → route by the confirmed value.

4 · Consent gate

Original released only if member opt-in AND admin release — both default off.

Each runs end-to-end on mock data, no application code. A Code node owns every decision; the Claude node only advises. Flip a value live to see the safety branch route.

05 · POC (no-code)Aushang
Privacy, proven — not asserted

No raw image. No un-redacted PII. Anywhere external.

  • Local redaction, fail-closed — Presidio + spaCy + a German regex pack mask names/phones/emails/IBAN/birthdates before the one external AI call.
  • DB column REVOKE — raw-PII columns are revoked from members; even an admin's browser can't read them. Admin PII access is server-only by construction.
  • Double-gated photo consent — the original is shown only on member opt-in × admin release (both default off), via a short-TTL signed URL.
  • Child photos aren't retained — a weekly Rückblick (the type most likely to show children) has its raw original deleted at publish; members keep only the masked image.
  • Build-blocking secret scan — a CI + pre-commit guard fails the build if any secret reaches the client bundle or source.

08 · Risk matrixAushang
Compliance

EU AI Act: Limited risk · GDPR: no PII to the AI

EU AI Act — Limited risk

Walks the decision tree: not prohibited, not a safety component, no Annex III area (makes no decision about a person). Human-in-the-loop + transparency met. The Kita context is communication, not education-access.

GDPR — privacy by construction

Processes real personal data (a Kita board), so the design carries it: raw photos stay in EU infra, the LLM gets redacted text only, PII columns REVOKE'd, DPIA done on capturing a child-naming notice, residual risk LOW–MED. Self-service erasure.

The most consequential control — local redaction + human confirmation — is what keeps it Limited-risk.
