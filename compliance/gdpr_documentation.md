# GDPR Documentation — SCM Master

**System:** AI demand forecasting + procurement copilot · **Regulation:** Regulation (EU) 2016/679 (GDPR)

> **Key fact up front:** SCM Master is a **B2B procurement/asset system**. Its core data —
> products, suppliers (companies), purchase orders, serial-tracked assets, demand history,
> commodity prices — is **not personal data**. The only personal data is a **small set of internal
> employee login accounts** and the **actor stamp** on audit logs. This sharply limits the GDPR
> surface, and the documentation below reflects that honestly rather than inventing exposure.

---

## 1. Data flow map

```
                    PERSONAL DATA (limited)                    NON-PERSONAL DATA (the bulk)
  ┌──────────────────────────────────────────┐   ┌─────────────────────────────────────────────┐
  │ Internal user accounts                     │   │ Products · Suppliers (companies) · POs ·     │
  │  • email, name, role, hashed password      │   │ Assets (serials) · demand history ·          │
  │ Audit actor stamp                          │   │ commodity prices · capacity                  │
  │  • "decided_by = <email/'agent'>"          │   │                                              │
  └───────────────┬────────────────────────────┘   └───────────────────┬─────────────────────────┘
                  │                                                      │
            collected at login / on a user action            generated by operations
                  │                                                      │
                  ▼                                                      ▼
        ┌───────────────────────────────────────────────────────────────────────┐
        │  SCM Master backend (FastAPI + Postgres, EU/Railway)                    │
        │  • authenticates, role-gates, stores hashed pw (bcrypt)                 │
        │  • logs who approved/placed each requisition (actor email)              │
        └───────────────┬───────────────────────────────────────────┬───────────┘
                        │ (no personal data sent)                     │ (no personal data sent)
                        ▼                                             ▼
        ┌───────────────────────────────┐               ┌──────────────────────────────┐
        │ Anthropic Claude API           │               │ Analytics cockpit (proxy)     │
        │  receives: procurement SIGNALS │               │  reads aggregate analytics    │
        │  (qty, price, lead time, …)    │               │  no per-person data           │
        │  NEVER receives user PII        │               └──────────────────────────────┘
        └───────────────────────────────┘
```

**The critical boundary:** user PII (emails, names) **never leaves** the backend — it is not sent to
the LLM, not sent to the cockpit. The LLM receives only **procurement signals** (quantities, prices,
lead times, supplier *company* names). The cockpit receives only **aggregate** analytics.

## 2. Processing activities register

| # | Data | Purpose | Legal basis | Retention | Third-party recipients |
|---|---|---|---|---|---|
| 1 | **User account** (email, name, role, hashed pw) | Authenticate users; enforce role-based access | **Contract** (Art. 6(1)(b)) — employment/system-access; legitimate interest for security | Duration of employment + 90 days, then deleted | None (stored in own Postgres) |
| 2 | **Audit actor stamp** (`decided_by` email/timestamp on approvals/placements) | Accountability — who approved which purchase (and SOX-style traceability) | **Legal obligation / legitimate interest** (Art. 6(1)(c)/(f)) — financial control | 7 years (financial-record norm), then anonymised | None |
| 3 | **Procurement signals** (qty, price, lead time, supplier *company* name) | Generate AI advice on a buy | **Legitimate interest** (Art. 6(1)(f)) — running procurement; **no personal data** | Not retained by the model (stateless call) | **Anthropic** (API) — non-personal data only |
| 4 | **Aggregate analytics** (spend by category/supplier, inventory, forecast) | Executive dashboard | **Legitimate interest** | Cached, refreshed; no personal data | Cockpit proxy (own infra) |
| 5 | **Operational data** (products, POs, assets, suppliers-as-companies) | Run the supply chain | **Legitimate interest / contract** | Business-record retention | None |

Activities 3–5 involve **no personal data** of natural persons (supplier names are *company* names).
Only #1 and #2 are personal-data processing, and both are low-sensitivity, internal, and minimal.

## 3. DPIA — highest-risk processing activity

**Selected activity: #2 — the audit actor stamp** (the only processing that links a named individual
to a consequential action — "who placed this purchase"). It's the highest-risk personal-data
processing because it's accountability data retained for years.

**Description of the processing:** When a user (or the agent on a user's authority) approves or
places a requisition, the system records `decided_by` (the user's email) and a timestamp on the
record, and appends a row to a decision log. Purpose: financial accountability and auditability.

**Necessity & proportionality:** Necessary — financial controls require knowing *who* authorised a
purchase; this is standard and expected in any procurement system. Proportionate — only email +
timestamp + the action are stored; no extra profiling, no special-category data, no automated
decision *about the person*. Data minimisation is respected (email is the minimum identifier).

**Risks to data subjects:**
| Risk | Likelihood | Impact |
|---|---|---|
| Unauthorised access exposes who bought what | Low | Low–Med (internal, commercially sensitive but not special-category) |
| Excessive retention beyond need | Med | Low |
| Function creep (using logs to evaluate staff performance) | Low | Med |

**Mitigation measures:**
- Access to logs is **role-gated** (ADMIN/finance only); the cockpit never shows per-person data.
- **Retention policy:** anonymise the actor field after the financial-record period (7 years).
- **Purpose limitation stated:** logs are for financial accountability, **not** staff performance
  monitoring — documented and enforced by access scope.
- Storage in EU-hosted Postgres; passwords hashed (bcrypt), never the actor data in plaintext logs
  beyond the email.
- Forge-locked production (no weak admin / no demo accounts) reduces breach surface.

**Residual risk rating: LOW.** The processing is minimal, internal, standard for procurement, and
mitigated. No high residual risk; no prior consultation with a supervisory authority required.

## 4. Data subject rights — how the system supports them

| Right (Art.) | Support |
|---|---|
| **Access (15)** | A user's account + their audit-stamp rows are queryable by email — an SAR can be fulfilled from Postgres. |
| **Rectification (16)** | Account fields (name/email) are editable by an admin via the user service. |
| **Erasure (17)** | Account deletion is supported; audit-stamp emails are **anonymised** (replaced with a non-identifying token) after the legal-retention period, satisfying erasure while preserving the financial record's integrity. |
| **Portability (20)** | Account + activity data is exportable as structured JSON/CSV (the API already serves JSON). |
| **Objection (21)** | A user may object to processing; since processing is contract/legal-obligation based (system access, financial control), objection is handled case-by-case — non-essential processing (if any) can be disabled per user. |
| **No solely-automated decisions (22)** | The system makes **no automated decision producing legal/significant effects on a person** — it decides about *purchases*, with a human in the loop, not about people. |

## 5. Third-party data transfers

| Service | Data sent | Personal data? | Legal mechanism | Where stored |
|---|---|---|---|---|
| **Anthropic (Claude API)** | Procurement signals (qty, price, lead time, supplier *company* names) | **No personal data** | Anthropic DPA / SCCs cover EU↔US transfer if invoked; **not needed for PII here since none is sent** | Not retained by the model (stateless inference) |
| **Railway (hosting)** | The application + database (which contains user accounts + audit stamps) | Yes (the DB) | DPA + EU region selection; SCCs for any non-EU sub-processing | EU region (configured) |
| **Power BI / cockpit** | Aggregate analytics only | **No personal data** | n/a (no PII transferred) | Own/EU infra |

**Transfer principle:** personal data (user accounts, audit stamps) stays within the EU-hosted
backend/DB under a DPA. The only external AI processor (Anthropic) receives **no personal data** —
the signal payloads are procurement facts about parts and companies, not natural persons. This is the
single most important GDPR property of the design and is enforced by the data-flow boundary in §1.
